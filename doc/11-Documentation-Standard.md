# 文档正式版标准

## 0. 文档控制

- 版本：v0.2
- 状态：Draft
- 用途：定义本文件的职责、范围与使用方式
- 适用对象：产品、设计、研发、测试、运维
- 单一来源：本文件
- 规范等级：MUST / SHOULD / MAY



## 1. 目的

本标准用于统一 Creative Production Runtime 文档的写法，确保文档可以直接被产品、设计、研发、测试和运维使用。

## 2. 适用范围

- 产品需求文档
- 交互规范文档
- 场景库
- 设计系统
- API / Contract
- 数据模型
- 测试策略
- 运行与排障手册
- 版本与发布策略
- 架构决策记录

## 3. 文档控制要求

每份正式文档都应包含：

- 标题
- 版本
- 状态
- 目标
- 范围
- 非目标
- 术语或引用
- 章节目录
- 变更记录

## 4. 规范性语言

### 4.1 MUST

- 必须遵守
- 不满足则视为不合格

### 4.2 SHOULD

- 强烈建议遵守
- 允许在明确说明原因时偏离

### 4.3 MAY

- 可选项
- 仅用于增强或未来扩展

## 5. 推荐章节模板

### 5.1 需求类文档

- 背景
- 目标用户
- 问题定义
- 场景
- 需求列表
- 验收标准
- 风险与约束

### 5.2 交互类文档

- 页面目标
- 页面组成
- 页面状态
- 关键交互
- 错误处理
- 空态与加载态

### 5.3 工程契约类文档

- 对象或接口定义
- 输入
- 输出
- 状态
- 错误
- 版本规则
- 测试要求

### 5.4 运行治理类文档

- 现象
- 排查步骤
- 恢复步骤
- 回滚步骤
- 风险提示
- 联系与升级

## 6. 命名规则

- 文档名称应能表达职责
- 章节标题应短且稳定
- 图示标题应描述布局或流程
- 表格标题应描述规则或矩阵

## 7. 图示规则

- 图必须服务于一个明确结论
- 图必须能被引用到对应章节
- 图名必须和正文标题一致或接近
- 图中的状态、节点、接口名应尽量复用统一术语

## 8. 验收标准

正式文档应满足：

- 读者能快速判断文档用途
- 读者能找到适用范围
- 读者能看到明确的规则和边界
- 读者能根据文档推进实现、测试或排障

## 9. 建议的文档协作方式

- 先写骨架，再补内容
- 先冻结术语，再扩展细节
- 先写规范，再写说明
- 先确定验收，再开始实现
