## 1. 产品目标

Creative Production Runtime / ScriptHub 是面向 AI 时代创意生产环境的跨 DCC 工具中心。它的目标不是替代 Hermes，也不是成为新的主聊天入口，而是在工具中心里管理和运行“可复用生产能力”。插件、脚本、Hermes 沉淀工具不是一级分类，而是同一种工具能力的不同来源或运行形态。用户点击某个工具后，ScriptHub 打开该工具的桌面级工具小窗，用于调参、确认风险和执行。

用户主要在外部 Hermes 中自然语言对话。第一次操作可能很费劲：路径错误、Connector 断开、Maya 选择为空、格式参数缺失、覆盖风险、权限问题。Hermes 负责反复尝试、追问、修复，直到任务跑通。ScriptHub 负责把这次跑通的经验记录成一个 Hermes 工具：它有参数、有运行历史、有产物、有安全确认、有 Hermes 修复记录，也能随着用户继续要求 Hermes 改进而升级。

同时，Hermes 写出的稳定 Maya / Blender / Unreal 插件或脚本也可以进入 ScriptHub。它们和 Hermes 跑通后沉淀出的个人小工具一样，都进入统一工具中心；区别只体现在能力分类、成熟度等级、运行环境和来源标签上。DCC 内只需要轻量 Connector，复杂的工具管理、参数面板、运行历史、安全确认和 Hermes 修复都留在 ScriptHub 的工具小窗中。

产品核心体验是：

```text
用户在 Hermes 里提出需求
  -> Hermes 反复执行、排错、修复直到跑通
  -> ScriptHub 生成“我的工具”
  -> 随着复用和验证，工具成熟度从草稿提升到可用、稳定、已验证或团队推荐
  -> 用户打开某个工具的桌面级置顶小窗，调整参数后再次运行
  -> 再次出错时 Hermes 继续介入修复
  -> 修复结果沉淀为工具的新能力或新版本
```

第一版个人用户默认看到的是：

- 工具中心：统一展示可复用生产能力，默认按能力分类、DCC 和成熟度组织。
- 我的工具：从 Hermes 成功操作中生成的可再次执行工具。
- 来源 / 运行形态：Hermes 沉淀、脚本、DCC 插件、外部服务或组合流程，仅作为标签展示。
- 成熟度：草稿、可用、稳定、已验证、团队推荐。
- 工具参数：输出路径、覆盖策略、导出范围、命名规则等可调整项。
- 运行历史：每次工具运行的输入、结果、产物、错误和修复过程。
- Hermes 修复：出错后 Hermes 如何介入、修了什么、是否增强了工具。
- 产物：本次操作生成的文件、版本和来源。
- 安全确认：执行前说明会读取什么、写入什么、是否覆盖文件、风险等级和确认结果。

团队和高级用户可以进一步看到 ToolCall、Trace、Audit、权限、版本、发布审核和评估数据。

## 2. 目标用户

### 2.1 个人用户

- 使用 Hermes 操作 Maya、Blender、Unreal、文件系统或生产工具的创作者。
- 经常重复执行导出、整理、转换、检查、发布等复杂操作的人。
- 不想反复写长提示词、不想看技术日志，但希望把 Hermes 跑通的复杂经验变成自己的工具的人。
- 不想在每个 DCC 内安装复杂插件面板，希望在 ScriptHub 工具中心打开桌面级置顶工具小窗来管理和运行工具的人。

### 2.2 团队用户

- 创意技术人员。
- Pipeline / TD。
- DCC 操作人员。
- 资产管理人员。
- 审批与管理角色。

## 3. 第一版要解决的问题

- 让用户继续在 Hermes 中完成意图表达、追问确认和任务执行。
- 让外部 Hermes 能通过 Tool Bridge 调用 ScriptHub 平台能力。
- 将 Hermes 的一次成功操作自动沉淀为可再次运行的工具。
- 支持 `script`、`dcc_plugin`、`external_service`、`composed_workflow` 等来源工具进入同一个工具中心，并通过统一参数面板和安全确认运行。
- 每个工具必须提供可调整参数，并说明会读取什么、写入什么、是否覆盖文件、需要什么权限、风险等级是什么。
- 执行失败后必须给出具体修复动作，而不是只展示错误。
- 执行失败后 Hermes 可以继续介入修复，修复结果应进入工具的运行历史和改进记录。
- 产物必须能追溯来源：由哪个 Hermes 会话、哪个工具、哪次运行、哪个输入对象生成。
- 个人用户默认不需要理解 ToolCall、Trace、Audit；这些作为高级排障和团队治理能力保留。

## 4. 典型场景

### 4.1 导出类

- 从 Maya 导出 FBX
- 批量导出多个资产
- 按命名规则输出到指定路径

### 4.2 处理类

- 批量重命名
- 批量烘焙 / 转换
- 批量检查并修复基础问题

### 4.3 流转类

- Maya 到 UE 的资产流转
- 生成结果资产并记录来源链
- 按审批规则发布或撤回

## 5. 非目标

- 不在 ScriptHub 内置完整自治 Agent
- 不把 ScriptHub 前端作为主对话入口
- 不做长期人格
- 不复制 Hermes 的长期记忆系统
- 不把 AI 模型绑定为唯一认知层
- 不要求用户在 ScriptHub 中手动完成常规任务创建和审批
- 不把个人用户默认界面做成工程师审计控制台

## 6. 成功标准

- Hermes 能通过 ScriptHub Tool Bridge 创建任务、查询状态、请求确认、注册产物和写入 trace。
- Hermes 完成一次复杂操作后，ScriptHub 能生成一个可再次运行的工具。
- 用户下次可以直接运行该工具，或调整参数后运行，而不需要重新描述完整长提示。
- 工具运行前必须显示：会读取什么、会写入什么、是否覆盖文件、风险等级、需要确认的动作。
- 用户能在外部 Hermes 对话中确认或拒绝高风险动作，也能在 ScriptHub 的安全确认层查看同一份确认信息。
- 用户能在 ScriptHub 中看到工具库、工具参数、运行历史、产物和 Hermes 修复记录。
- 用户能在同一个工具中心中按能力分类、DCC 和成熟度找到工具，并通过来源标签理解它来自 Hermes 沉淀、脚本、插件或外部服务。
- 用户能继续让 Hermes 完善某个工具，并看到工具版本或能力变化。
- 团队或高级用户能进一步追踪 ToolCall、Trace、Audit、权限和技能版本。

## 7. 第一版范围

### 7.1 个人默认层

- 工具中心：统一展示可复用生产能力，主筛选为能力分类、DCC 和成熟度。
- 我的工具：展示从成功操作生成的工具。
- 来源标签：展示 Hermes 沉淀、脚本、DCC 插件、外部服务或组合流程。
- 工具详情：展示工具参数、运行历史、Hermes 修复记录和版本。
- 参数编辑：允许用户调整输出路径、覆盖策略、导出范围、命名规则等参数后再次运行。
- 桌面级工具小窗：用户在工具中心点击某个工具后打开独立置顶小窗，修改参数、确认风险并执行，不依赖 DCC 内置复杂 UI。
- 安全确认：展示读写范围、覆盖风险、权限和确认结果。
- 产物架：展示生成文件、版本、来源和打开位置。
- Hermes 修复：展示失败原因、Hermes 介入过程、可执行修复动作和工具增强结果。

### 7.2 高级与团队层

- Agent Activity Console / Hermes 活动镜像。
- ToolCall Timeline。
- 任务中心和任务详情页。
- 工作流页。
- 能力库。
- 资产库。
- 连接器面板。
- Trace / Review / Approval。
- 事件与审计。

## 8. 明确暂缓

- 多 Agent 协同
- 自我演化
- 复杂长期记忆
- 自动生成完整工作流的高阶优化

## 9. 需求拆解原则

### 9.1 需求必须可验收

- 每一条需求都要有可观察结果
- 每一条需求都要能写出验收条件
- 每一条需求都要能指向具体页面或能力

### 9.2 需求必须可分层

- 产品目标
- 用户价值
- 场景
- 页面
- 组件
- API

### 9.3 需求必须可追踪

- 每个需求都应有唯一编号
- 每个需求都应关联到对应的任务、图示或契约
- 每个需求都应标明是否属于第一版

## 10. 第一版验收标准

- 外部 Hermes 能成功调用 ScriptHub 工具创建一个任务
- 用户能在 Hermes 对话中看到计划和确认请求
- 用户能在 Hermes 对话中对高风险动作进行审批
- 用户能在 ScriptHub 中看到由这次成功操作沉淀出的工具
- 用户能在 ScriptHub 中看到能力分类、DCC、成熟度和来源标签，而不是被迫理解“插件 / Hermes 工具”这种二分分类。
- 用户能修改工具参数并再次运行
- 再次运行失败时，用户能让 Hermes 继续修复并把结果记录回工具
- 用户能在任务详情里看到状态流和 Trace
- 用户能在资产页看到来源链和版本
- 用户能看到 Connector 当前健康状态
- ScriptHub 前端默认不要求用户手动点击执行主流程

## 11. 风险与约束

- 不能把 AI 计划直接当执行结果
- 不能把脚本直接当能力资产
- 不能把错误只当弹窗
- 不能把结果只当文件
- 不能把 UI 只当页面集合
